- 已完成项目需求文档、任务文档、技术选型文档整理

- 已完成核心数据模型设计、`SQLAlchemy` 模型与 `Alembic` 初始迁移

### 1.2 进行中

- 前端多角色页面仍以 mock 数据联调为主，正在逐步切换到 `apps/backend-fastapi` 真实业务 API
- 管理端规则、词库、模板等页面已具备列表展示，但仍缺少新增、编辑、启停等完整闭环
- 社区工单、子女端通知、老人端提醒等页面已可演示，但与识别服务和自动流转链路尚未打通
- 后端业务接口当前以演示数据服务为主，尚未全面切换为数据库读写实现

### 1.3 未完成

- 风险识别规则、词库、检测服务尚未落地
- 业务接口到数据库的真实读写闭环尚未全面完成
- `Redis`、消息队列、实时通知等基础设施尚未接入
- 风险识别到通知、工单、处置的主链路尚未真正打通
- 联调、测试、验收文档与演示流程尚未完成

## 2. 总体待办原则

- 优先完成 V1 最小闭环，不扩展高精度模型和重型能力
- 优先做“能联调、能演示、能验收”的真实功能，而不是继续堆静态页面
- 优先打通“识别 -> 告警 -> 通知 -> 工单 -> 处置 -> 留痕”主流程
- 所有页面开发应尽量对应真实接口和真实数据结构，减少后续返工

## 7. 当前建议的开发顺序

1. 先实现规则版风险识别服务，补齐词库、检测逻辑和统一风险结果结构。
2. 再打通“识别 -> 告警 -> 通知 -> 工单 -> 处置”主链路。
3. 接着把当前核心前端页面从 mock 数据切换到真实业务 API。
4. 然后补齐管理端规则配置、社区工单处置等剩余闭环能力。
5. 最后完善 `Redis`、实时通知、联调测试和验收演示材料。
