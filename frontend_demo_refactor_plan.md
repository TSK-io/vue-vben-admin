# 桑榆智盾前端改造工程蓝图

更新时间：2026-04-27

## 1. 核心共识

本轮改造的核心共识是：

> 以“电话号码”为系统身份骨架，用 Web 模拟电话和短信通讯，让老人、输入端、子女、社区 / 警察在同一套系统里完成真实感更强的反诈剧情闭环。

本系统不再优先建设独立 `/demo` 演示沙盘。后续主线是直接改造现有角色端，让团队成员分别登录自己的角色，通过电话和短信事件完成剧情演示。

系统最终要呈现的是一个多角色协同的反诈守护产品，而不是一个临时演示页。

## 2. 角色和身份结构

系统角色定义为：

```text
老人端：被守护者，拥有自己的电话号码和类手机操作系统
输入端：外部通信输入器，登录后可自定义本次发起号码，用于模拟安全号码、陌生号码和诈骗号码
子女端：安全联系人 / 家人守护者，拥有自己的电话号码和家人管理界面
社区 / 警察端：所有风险事件的汇总 dashboard 和处置工作台
管理员端：沿用当前后台风格，保留系统配置、用户、规则和记录管理
```

电话号码是后续剧情和业务逻辑的关键标识：

- 老人、子女都应有自己的固定电话号码。
- 老人端通过电话号码管理安全联系人、黑名单和绑定关系。
- 输入端账号本身是操作身份，不直接等同于剧情号码；每次通信事件由输入端填写或选择本次发起号码。
- 输入端通过“本次发起号码 + 目标老人号码”定向拨打老人、发送短信。
- 子女端通过电话号码被老人绑定为安全联系人，也用于联系和通知。
- 社区 / 警察端通过电话号码追踪事件来源、被守护者、家属联系人和处置记录。

## 3. 多角色剧情演示方式

比赛现场不再只展示单页演示中心，而是让团队成员分别登录自己的角色：

```text
团队成员 A：老人端
团队成员 B：输入端
团队成员 C：子女端
团队成员 D：社区 / 警察端
团队成员 E：管理员端，可选
```

电话剧情主线：

```text
输入端选择或填写本次发起号码
-> 输入老人端电话号码
-> 拨打诈骗电话
-> 老人端像手机一样收到陌生来电
-> 老人接听，双方通过 Web 模拟通话
-> 通话结束后进入后端识别
-> 后端返回风险等级、诈骗类型、证据片段和建议动作
-> 老人端收到风险提示弹窗
-> 若判定为诈骗，系统自动将该来电号码加入老人端黑名单
-> 子女端收到风险通知，并可查看通话事件详情
-> 社区 / 警察端 dashboard 汇总事件并进入处置流程
```

短信剧情主线：

```text
输入端选择或填写本次发起号码
-> 输入老人端电话号码
-> 发送诈骗短信
-> 老人端像手机短信 App 一样收到陌生短信
-> 陌生号码首次短信进入后端模型推理
-> 后端返回风险等级、诈骗类型、证据片段和建议动作
-> 老人端收到风险提示弹窗
-> 若判定为诈骗，系统自动将该短信号码加入老人端黑名单
-> 子女端收到风险通知，并可查看短信事件详情
-> 社区 / 警察端 dashboard 汇总事件并进入处置流程
```

## 4. 老人端定义

老人端登录后应是一个类似手机操作系统的界面，而不是后台页面。

老人端必须具备自己的电话号码。

核心能力：

- 拨打电话
- 接听电话
- 接收短信
- 发送短信
- 管理安全联系人
- 管理黑名单
- 绑定 / 解绑子女端
- 查看通话事件和短信事件
- 接收后端判定结果的风险提示弹窗

安全联系人通常包括子女、老伴、朋友和其他可信联系人。

黑名单用于存放已经被系统确认或老人手动标记的风险号码。电话和短信场景中，一旦后端判定陌生号码为诈骗类型，系统应自动将该号码加入老人端黑名单。

核心页面建议：

```text
/elder/home       手机首页 / 守护桌面
/elder/phone      电话、拨号、来电、通话中
/elder/messages   短信会话、短信详情、风险短信提醒
/elder/contacts   通讯录、安全联系人、黑名单
/elder/bindings   子女端绑定 / 解绑
/elder/alerts     风险记录
<!-- /elder/help：原计划 SOS 紧急求助入口，建议取消（第一阶段不纳入开发范围）。 -->
```

UI 方向：

- 像手机操作系统，而不是后台。
- 使用顶部状态栏、手机桌面卡片、底部 Tab。
- 大字号、大图标、大按钮，适合老人使用。
- 电话页面支持拨号、陌生来电、接听、拒接、通话计时和挂断。
- 短信页面支持会话列表、短信详情、陌生短信风险标记。
- 通讯录页面明确区分安全联系人和黑名单。
- 风险提醒语言直接，例如“疑似诈骗电话，请不要转账，已通知家属”。

## 5. 输入端定义

输入端是外部通信事件的发起和呼入模拟器，用于模拟不同外部号码对老人发起电话和短信。

输入端不是普通业务用户，也不代表一个固定“诈骗号码”。团队成员登录输入端账号后，可以操控本次通信事件的发起号码，用来验证系统对安全号码、陌生号码、已拉黑号码和诈骗号码的不同反应。

因此需要区分两类身份：

```text
operator_user_id：登录输入端的系统账号，用于权限和操作留痕
source_phone：本次电话或短信事件展示给老人端、并参与安全判断的外部号码
target_phone：目标老人端电话号码
```

输入端可以包装成简陋版类手机操作系统，但本质是剧情输入端，只保留剧情需要的最小能力。

核心能力：

- 选择或填写本次发起号码
- 支持安全库号码、陌生号码、已拉黑号码、诈骗号码等测试场景
- 输入老人端电话号码
- 定向拨打诈骗电话
- 定向发送诈骗短信
- 查看本次事件是否送达、是否触发推理、是否触发拉黑和通知

核心页面建议：

```text
/input/phone      发起号码设置、目标号码输入、拨打电话
/input/messages   发起号码设置、目标号码输入、发送短信
```

UI 方向：

- 简陋、直接、工具化。
- 不需要复杂管理能力。
- 不预先选择诈骗类型。
- 电话内容由团队成员真人扮演。
- 短信内容由团队成员输入。
- 系统根据后端识别结果决定是否诈骗、属于哪类诈骗。
- WebRTC 或 Web 通话连接仍使用登录输入端的系统账号；老人端看到和系统判断的是 `source_phone`。

## 6. 子女端定义

子女端是安全联系人和家人守护端，也必须拥有自己的电话号码，方便老人端添加、绑定和联系。

核心能力：

- 查看“我的家人”
- 管理和查看绑定老人
- 查看老人详细通讯信息
- 查看老人短信详情信息
- 查看通话事件
- 查看陌生电话号码来电事件
- 查看陌生号码短信事件
- 接收后端判定结果的风险提示
- 联系老人
- 查看风险事件处置进度

重点页面：

```text
/family/overview       家人守护首页
/family/seniors        我的家人
/family/seniors/:id    家人详情：通讯、短信、事件、风险记录
/family/alerts         风险详情和模型证据
/family/notifications  通知记录
```

UI 方向：

- 更像家人守护 App / Web 工作台。
- 首页突出“我的家人是否安全”。
- 家人详情页能看到老人通讯和短信风险事件。
- 最近动态清楚展示陌生来电、陌生短信、风险告警。
- 提供直接操作：联系老人、发送提醒、查看风险详情、标记已处理。

## 7. 社区 / 警察端定义

社区 / 警察端是所有事件角色的汇总 dashboard，供社区人员查看、管理、联系被守护者和子女。

核心能力：

- 查看辖区风险总览
- 查看所有电话风险事件
- 查看所有短信风险事件
- 查看老人、子女、陌生号码之间的关联
- 查看后端模型判定结果
- 查看自动黑名单记录
- 联系被守护老人
- 联系子女
- 管理和推进处置状态

重点页面：

```text
/community/dashboard   风险汇总 dashboard
/community/workorders  风险处置工单
/community/seniors     被守护者列表
/community/reports     风险统计报表
```

UI 方向：

- 保留管理系统的高信息密度。
- 使用清晰的侧边导航、统计卡、趋势图、风险分布、工单列表。
- 风险事件按优先级突出。
- 工单详情展示完整链路：触发号码、老人号码、子女号码、模型判断、证据、通知状态、黑名单动作、处置状态。

## 8. 管理员端定义

管理员端沿用当前后端管理风格，不作为第一阶段视觉改造重点。

保留能力：

- 用户管理
- 角色权限
- 风险规则
- 内容管理
- 系统配置
- 事件记录查询

管理员端第一阶段只做必要的视觉统一和入口补齐，不抢老人端、输入端、子女端和社区端的主线优先级。

## 9. 风险识别和自动黑名单规则

电话规则：

```text
陌生号码来电
-> 老人端接听并完成 Web 模拟通话
-> 通话结束后送入后端识别
-> 后端判定风险等级和诈骗类型
-> 老人端弹窗提示
-> 子女端通知
-> 社区 / 警察端记录
-> 若判定为诈骗，自动将来电号码加入老人端黑名单
```

短信规则：

```text
陌生号码短信
-> 老人端收到短信
-> 只要是陌生电话号码首次发来的短信，就送入后端模型推理
-> 后端判定风险等级和诈骗类型
-> 老人端弹窗提示
-> 子女端通知
-> 社区 / 警察端记录
-> 若判定为诈骗，自动将短信号码加入老人端黑名单
```

安全联系人和黑名单规则：

- 安全联系人号码的电话和短信默认视为可信通讯，不走诈骗推理主链路。
- 黑名单号码再次来电或发短信时，应在老人端直接突出拦截或强风险提示。
- 陌生号码不是立即拉黑，必须经过后端判定后才能自动进入黑名单。
- 老人端应保留手动添加 / 移除黑名单的能力。

## 10. 后端使用原则

后端当前能力先不大改，继续沿用已有接口：

```http
POST /api/v1/risk-recognition/call-audio
POST /api/v1/risk-recognition/call
POST /api/v1/risk-recognition/sms
GET  /api/v1/risk-alerts
GET  /api/v1/notifications
GET  /api/v1/community/workorders
POST /api/v1/community/workorders/{workorder_id}/transition
```

如果现有后端尚未覆盖电话号码、联系人、黑名单和 Web 通讯事件状态，第一阶段可以先通过前端状态和轻量接口适配完成演示闭环；后续再补正式数据模型。

第一阶段不重写后端模型识别逻辑。

## 11. 第一阶段推荐开发顺序

```text
1. 定义电话号码、联系人、黑名单、通讯事件的前端数据结构
2. 重做 /elder/home：手机系统式守护首页
3. 新增 /elder/phone：拨号、来电、接听、挂断、通话识别入口
4. 新增 /elder/messages：短信列表、短信详情、陌生短信识别入口
5. 新增 /elder/contacts：安全联系人、黑名单、子女绑定入口
6. 新增 /input/phone：输入端发起号码设置、目标号码输入、拨打电话
7. 新增 /input/messages：输入端发起号码设置、目标号码输入、发送短信
8. 改造 /family/overview：家人守护首页
9. 改造 /family/seniors：我的家人和家人详情
10. 改造 /family/alerts：风险详情和模型证据
11. 改造 /community/dashboard：事件汇总 dashboard
12. 改造 /community/workorders：工单处置链路
```

## 12. 第一阶段验收标准

- 老人、子女两个业务身份都有自己的固定电话号码；输入端每次事件可自定义发起号码。
- 老人端第一眼看起来像手机操作系统，而不是后台。
- 输入端可以用自定义发起号码给指定老人号码拨打电话和发送短信。
- 老人端可以接听电话、接收短信、管理安全联系人和黑名单。
- 陌生短信首次进入后端推理逻辑。
- 陌生电话通话结束后进入后端推理逻辑。
- 后端判定为诈骗后，对应号码能自动进入老人端黑名单。
- 老人端能收到电话和短信风险提示弹窗。
- 子女端能查看绑定家人的通讯事件、短信事件和风险判断结果。
- 社区 / 警察端能汇总所有风险事件，并支持联系老人和子女。

## 13. 后续增强方向

第一阶段完成后，再考虑增强：

- 真实 WebRTC 或更稳定的 Web 通话模拟
- 演示数据一键重置
- 号码库和官方可信号码
- 更完整的黑名单同步
- 子女端短信提醒或语音提醒
- 社区端风险热力图
- 管理端风险统计面板
- 比赛演示脚本和讲解页

## 14. 最终产品表达

本项目最终要呈现的是：

> 桑榆智盾是一个面向老人群体的本地化反诈守护系统原型。系统通过 Web 模拟有电话号码身份的手机端短信和电话场景，通过本地反诈模型完成真实风险识别，并将识别结果转化为老人提醒、子女通知、社区工单和后台记录，形成从风险发现到协同处置的完整闭环。

核心不是替代国家反诈 App，而是作为老人群体的补充防线。

核心不是监听所有通话，而是围绕陌生通话和可疑短信做低侵入、可解释、可协同的风险守护。
