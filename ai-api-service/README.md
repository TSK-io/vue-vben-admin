# AI API Service

## 模块定位

`ai-api-service` 是一个放置在仓库根目录的独立后端模块，用于提供 AI 相关的 API 服务。

这个模块的设计目标是：

- 完全独立于当前 `vue-vben-admin` 主项目
- 可以单独开发、单独部署、单独运维
- 未来部署在另一台服务器上
- 不依赖当前仓库内的 `apps`、`packages`、`internal` 等任何现有模块

当前阶段先完成目录占位与设计说明，后续再根据选型补充实际服务代码。

## 设计原则

- 独立仓库思维：虽然目录位于当前仓库根目录，但应按“外部服务”来设计
- 零耦合：不要直接引用主项目中的组件、工具函数、类型、配置或构建体系
- 可替换：底层可以根据需要接入 OpenAI、Azure OpenAI、阿里云百炼、火山方舟或自建模型网关
- 可部署：默认考虑 Linux 服务器、Docker 容器、反向代理、环境变量配置
- 可扩展：后续可逐步加入鉴权、限流、日志、配额、模型路由、对话上下文管理等能力

## 与主项目的边界

这个模块和前端主项目之间建议通过 HTTP/HTTPS API 通信，不共享运行时、不共享构建产物、不共享内部代码。

建议边界如下：

- 前端项目只负责调用该服务暴露出的接口
- AI 服务只负责模型调用、提示词编排、流式输出、结果封装与安全控制
- 双方通过明确的接口协议协作，例如 REST、SSE 或 WebSocket

## 推荐职责范围

这个模块后续可以承担以下职责：

- 提供统一的 AI 对话接口
- 提供文本生成、摘要、翻译、改写等能力
- 提供流式响应接口
- 统一管理模型提供商配置
- 处理 API Key、签名、权限校验、限流与审计日志
- 为前端屏蔽不同模型平台之间的调用差异

## 暂不纳入主工程

为了保证它的隔离性，当前建议保持以下约束：

- 不加入根目录 `pnpm-workspace.yaml`
- 不复用主项目的 ESLint、TSConfig、构建脚本和发布流程
- 不直接引用主项目中的代码
- 不让主项目构建流程默认扫描或打包这个模块

这样做可以确保它未来拆分到独立仓库或迁移到其他服务器时成本最低。

## 后续建议

下一步如果要继续落地，建议按下面顺序推进：

1. 确定技术栈，例如 `Node.js + Fastify`、`Node.js + NestJS` 或 `Go`
2. 明确首批接口，例如 `/health`、`/chat`、`/stream-chat`
3. 设计环境变量与密钥管理方案
4. 补充基础工程文件，例如 `package.json`、`.gitignore`、`.env.example`
5. 增加 Docker 部署配置，方便未来部署到独立服务器
