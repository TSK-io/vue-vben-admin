# 银盾 IM 项目需求文档

## 1. 项目重定义

本项目不再按“多个前台门户 + 一个后台”的思路推进，而是调整为以下产品结构：

- 用户端主产品：`silver-shield-mobile`
  基于 `uni-app` 开发的即时通讯应用，优先支持 H5 开发调试，后续迁移到微信小程序验证。
- 管理端：`apps/web-antd`
  仅作为后台管理系统使用，不再承载老年端、家属端、社区端等前台用户页面。
- 主业务后端：`apps/backend-fastapi`
  提供账号、关系、会话、消息、通知、告警、内容配置、后台管理等业务 API。
- AI 风险服务：`ai-api-service`
  作为独立服务提供消息文本、聊天记录、链接等诈骗风险识别能力。

一句话定位：

> 这是一个“以即时通讯为核心、以风险提醒和守护能力为差异化价值”的 `uni-app` 用户端项目，`web` 只负责后台配置、审核与运营。

## 2. 产品目标

### 2.1 核心目标

- 让老年用户与家属/守护人之间可以像使用普通聊天软件一样进行低门槛沟通。
- 把“诈骗风险提醒、可疑消息识别、一键求助”作为 IM 过程中的增强能力，而不是把用户端做成复杂的治理平台。
- 让后台管理人员可以在 `web` 端完成账号、关系、风控、内容和系统配置管理。

### 2.2 MVP 目标

首个可交付版本需要满足以下闭环：

1. 用户可登录移动端并进入自己的会话列表。
2. 用户可查看联系人、进入单聊、发送文本消息。
3. 系统可对可疑文本或链接做风险提示。
4. 用户可在风险场景下快速发起求助。
5. 后台可查看用户、关系、风险事件和基础配置。

## 3. 角色定义

### 3.1 用户端角色

- 老年用户
  重点关注简洁、可读、可操作，核心使用场景为聊天、查看提醒、语音联系、紧急求助。
- 家属/守护人
  重点关注会话、风险提醒、老人状态关注、求助响应和守护设置。

说明：

- 两类角色可以继续放在同一个 `uni-app` 工程内实现。
- 是否保持“一账号双角色”能力，可以在后端账号体系稳定后再决定；MVP 阶段允许按账号角色直接分流。

### 3.2 后台角色

- 超级管理员
  管理账号、角色、权限、系统参数。
- 运营管理员
  管理用户资料、关系绑定、内容配置、公告与帮助内容。
- 风控审核员
  查看风险识别记录、复核结果、处理高风险事件。
- 客服/支持人员
  查看求助记录、通知状态、处理人工介入流程。

## 4. 产品边界

### 4.1 用户端必须承载的能力

- 登录与身份识别
- 联系人/守护关系
- 会话列表
- 单聊消息页
- 风险提示
- 一键求助
- 通知提醒
- 适老化设置

### 4.2 后台必须承载的能力

- 后台账号登录与权限控制
- 用户与角色管理
- 联系人与绑定关系管理
- 风险事件与识别结果管理
- 内容管理
- 系统参数与风控规则管理
- 日志、审计与统计报表

### 4.3 明确不再做的方向

- `web` 不再作为老年端、家属端、社区端的前台使用入口。
- `web` 不再继续扩写“前台门户式”菜单结构。
- 用户端不做成复杂的综合治理平台首页，主界面必须围绕会话、联系人和消息展开。
- IM 初期不追求完整企业微信/微信级能力，优先完成单聊、提醒和求助闭环。

## 5. 核心业务流程

### 5.1 主链路

1. 用户登录移动端。
2. 进入会话列表，查看未读消息和重要提醒。
3. 打开单聊页，发送或接收文本消息。
4. 系统对消息文本或链接进行风险识别。
5. 若命中中高风险，前端展示风险提示，并可引导用户发起求助或通知守护人。
6. 后台查看风险事件、复核结果和处置记录。

### 5.2 求助链路

1. 老年用户在聊天页、风险页或首页发起一键求助。
2. 主业务系统记录求助事件并通知相关联系人。
3. 守护人可在移动端查看求助详情。
4. 后台可查看求助记录、处理状态和人工介入结果。

### 5.3 管理链路

1. 管理员登录 `web` 后台。
2. 管理账号、关系、内容、规则和系统参数。
3. 查看风险识别记录和消息风险复核结果。
4. 处理投诉、审核和处置留痕。

## 6. 用户端需求

### 6.1 登录与账号

- 支持账号登录。
- 支持基础登录态缓存与自动恢复。
- 支持按用户角色进入不同首页或主导航。
- 支持退出登录、重新登录和异常态兜底。

MVP 建议：

- 先做手机号/账号 + 验证码或密码登录中的一种。
- 登录后直接拉取当前用户信息、角色、联系人和最近会话。

### 6.2 联系人与关系

- 展示联系人列表。
- 展示老人 - 家属/守护人绑定关系。
- 支持查看联系人基础资料。
- 支持进入单聊。

可后置能力：

- 联系人搜索
- 联系人备注
- 邀请绑定/扫码绑定

### 6.3 会话列表

- 展示最近会话。
- 展示未读数、最后一条消息、时间、风险标记。
- 支持下拉刷新和空态展示。
- 支持从通知或求助记录跳转到相关会话。

### 6.4 聊天详情

- 支持文本消息发送与接收。
- 支持消息状态展示，如发送中、发送成功、发送失败。
- 支持历史消息分页拉取。
- 支持高风险消息样式强调。
- 支持对可疑链接做风险提示。

当前阶段不强求：

- 群聊
- 文件消息
- 图片/视频大规模上传
- 完整表情体系

### 6.5 风险提醒

- 对输入文本、接收文本、聊天记录或链接提供风险识别。
- 根据识别结果展示 `low / medium / high` 风险等级。
- 在聊天页、风险页、通知页可查看风险原因和建议。
- 高风险时支持快捷跳转求助。

设计原则：

- 风险提醒是 IM 的辅助能力，不能压过“聊天”本身。
- 风险弹层和提示要足够克制，避免影响正常消息使用。

### 6.6 语音通话

- 保留通话入口和通话记录页。
- MVP 阶段允许先只做页面、状态和日志记录。
- 实时语音能力可在 P2 再接入第三方音视频 SDK。

### 6.7 一键求助

- 首页、聊天页、风险提醒页都应有求助入口。
- 支持提交求助类型、备注信息和上下文来源。
- 支持展示求助成功页与历史记录。
- 支持守护人查看求助详情。

### 6.8 通知与设置

- 展示系统通知、风险提醒、求助回执。
- 提供字体大小、颜色对比度、提示音等适老化配置。
- 提供账号与隐私设置入口。

## 7. 后台需求

### 7.1 后台定位

`apps/web-antd` 的唯一定位是后台管理系统，其主要服务对象是管理员、运营人员、风控审核员和客服支持人员。

后台不再承载：

- 老年端首页
- 家属端首页
- 社区端前台工作台

如确有社区/客服工作需求，也应作为后台角色和后台菜单实现，而不是再次做一个前台门户。

### 7.2 后台模块

### 账号与权限

- 管理员登录
- 角色管理
- 权限点管理
- 操作日志

### 用户与关系

- 用户列表
- 用户资料查看
- 联系人/绑定关系管理
- 黑名单或异常账号处理

### 风险中心

- 风险识别记录列表
- 风险详情
- 人工复核
- 高风险事件处置状态

### 内容与配置

- 帮助内容
- 反诈宣教内容
- 风险提示文案
- 系统参数
- AI 提示词版本和规则开关

### 统计与审计

- 消息量与活跃度统计
- 风险事件统计
- 求助事件统计
- 审计日志与导出

## 8. 后端与服务边界

### 8.1 主业务后端 `apps/backend-fastapi`

负责：

- 登录鉴权
- 账号、角色、权限
- 用户资料与联系人关系
- 会话列表与消息存储
- 通知、求助、风险事件记录
- 后台管理 API

建议新增或明确的业务域：

- `/api/mobile/auth/*`
- `/api/mobile/contacts/*`
- `/api/mobile/conversations/*`
- `/api/mobile/messages/*`
- `/api/mobile/alerts/*`
- `/api/mobile/sos/*`
- `/api/admin/*`

### 8.2 AI 服务 `ai-api-service`

负责：

- 单条文本风险识别
- 聊天记录批量识别
- 链接风险识别
- 人工复核记录接口

原则：

- AI 服务不直接承担 IM 主业务逻辑。
- AI 服务保持独立部署、独立扩展、独立容错。
- 用户端尽量通过主业务后端调用 AI 服务，避免前端直接依赖复杂 AI 接口。

## 9. 技术与架构要求

### 9.1 用户端技术策略

- 使用 `uni-app + Vue 3 + Pinia`。
- 先以 H5 为主要开发与调试载体。
- 当整体功能完成度达到约 95% 时，再集中迁移到微信小程序做专项兼容验证。

### 9.2 IM 实现策略

- P1 优先做“业务系统代管的文本聊天 MVP”。
- 消息实时性在 P1 可采用短轮询或主动刷新。
- P2 再评估 `WebSocket` 或第三方 IM/RTC 能力。

### 9.3 后台技术策略

- 基于现有 `vue-vben-admin` 工程继续收敛为后台管理系统。
- 现有混杂的前台路由、菜单、角色命名需要逐步清理。
- 后台 UI 优先强调配置效率、检索效率、审计能力，而不是用户端体验。

## 10. 非功能要求

- 安全性：登录态、接口鉴权、敏感字段脱敏、操作留痕。
- 稳定性：AI 服务失败时必须可降级，不影响聊天主流程。
- 可观测性：前端异常、接口错误、识别失败、求助失败要可追踪。
- 可维护性：用户端、后台、主业务后端、AI 服务边界清晰。
- 适老化：大字号、高对比度、简洁交互、关键按钮易点击。

## 11. 里程碑建议

### P0 方向收口

- 明确产品边界和角色定义。
- 根文档、README、菜单结构统一口径。
- `web` 从前台门户思路收敛为后台管理。

### P1 IM MVP

- 登录
- 联系人
- 会话列表
- 单聊
- 风险提醒
- 一键求助

### P2 管理与风控增强

- 后台风险中心
- 内容配置
- 角色权限
- 更完整的消息审核与统计

### P3 实时与跨端增强

- 更稳定的实时消息机制
- 语音通话 SDK
- 微信小程序专项适配

## 12. 当前结论

当前项目的正确方向应是：

- `silver-shield-mobile` 做用户真正使用的即时通讯软件。
- `apps/web-antd` 只做管理后台。
- `apps/backend-fastapi` 做主业务服务。
- `ai-api-service` 做独立的风险识别与审核支撑服务。

后续所有页面规划、接口设计和任务拆解，都必须围绕这个边界执行，避免再次把后台写成前台、把 IM 写成综合门户。
